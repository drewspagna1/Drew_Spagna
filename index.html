Agentic Workflow Automation

Durable orchestration pipeline for AI-driven work processing

I didn't build this orchestration framework from scratch, I inherited it and pushed it a lot further. The core pattern processes work items end to end without a person babysitting every step: a durable orchestrator kicks off a get-work stage, chains into a second activity that loads validated items onto a queue, and a separate queue-triggered function actually does the work asynchronously, so one slow item can't block the whole batch. My contribution was taking that framework and building a standardized, object-oriented orchestration layer on top of it, as an optional path teams could opt into instead of rebuilding the same pipeline logic by hand for every new customer. Nobody was forced onto it, but it was there so a team could apply the blueprint and adapt just the pieces specific to that customer's process instead of starting from a blank file. Every completed item posts back through its own validation queue, so bad postbacks get caught and retried instead of silently disappearing. Failures aren't treated as one bucket either: business exceptions, things where the automation genuinely can't make the call, route to a human review queue, while system exceptions, timeouts, connectivity issues, that kind of thing, get retried or alerted on automatically. I also built in a standing health check that monitors database, third-party API, and auth connectivity so the app can flag or reset itself before anyone notices it's degraded, plus a manual review view so a person can close out or reprocess anything the automation couldn't resolve on its own.

RPA-to-AI Migration

Modernizing legacy RPA into LLM-assisted automation

A long-running RPA program had dozens of bots that were great at structured, rules-based work and fell apart on anything needing judgment: unstructured documents, ambiguous free text, exceptions the decision tree just wasn't built for. Instead of ripping it all out, I scoped a migration path that layered LLM reasoning on top of the existing automation, only for the tasks where rules-based logic was actually breaking, and left the reliable bots doing what they were already good at. The real work was deciding exactly where "deterministic automation" should end and "AI judgment" should start, then building clean interfaces between the two so the whole pipeline stayed auditable.

ApproachHybrid RPA + LLM architecture, scoped modernization plan
StackBlue Prism, Python, LLM APIs, document/text processing
OutcomeExtended automation coverage into previously manual exception handling
Technical Integration Proposal

Keeping Jira the single source of truth across three systems

This one's a proposal I put together for an interview exercise, not delivered client work, but it's close to how I'd actually approach the problem so it's worth showing in full. The scenario: a prospective customer wanted to hand off global hardware procurement and lifecycle management to a new IT platform, but their Jira Asset Management instance had to stay the authoritative record auditors query, and it couldn't step on the Mac and Windows device management tools already live. I designed and proposed an integration architecture where the new platform would drive procurement and lifecycle events, the device management tools would report real-time device health, and Jira would reconcile all three into one conflict-resolved record. The proposal included a full field-mapping spec, explicit conflict-resolution rules for when sources disagree (a stale device check-in doesn't get to overrule an active repair status), a PII-stripping layer before anything writes to Jira, and a phased rollout plan that starts read-only and upgrades to real-time sync without re-architecting later. I also built a small local mock server to test my own field-mapping and conflict-resolution assumptions against a simulated version of the platform's API before presenting it, rather than presenting a diagram I hadn't stress-tested at all.

ApproachMulti-system reconciliation architecture, field mapping, conflict-resolution rules, phased rollout plan
StackREST APIs, delta polling, webhook-based event sync, PII-filtering middleware
OutcomeA complete, defensible technical proposal, self-validated against a mocked API, not a shipped integration

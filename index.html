Agentic Workflow Automation

Durable orchestration pipeline for AI-driven work processing

Problem

Every new customer engagement rebuilt the same durable-orchestration pipeline by hand — get-work, queueing, async processing, exception handling — from a blank file.

Approach

I inherited the core framework and layered a standardized, object-oriented orchestration template on top, as an opt-in path teams could adopt instead of rebuilding. Work flows end to end without babysitting: a durable orchestrator triggers a get-work stage, loads validated items onto a queue, and a separate queue-triggered function does the work asynchronously, so one slow item can't block the batch. Completed items post back through their own validation queue, so bad postbacks get caught and retried instead of silently disappearing.

Reliability

Failures split by type: business exceptions — where the automation can't confidently make the call — route to a human review queue, while system exceptions like timeouts and connectivity drops retry with backoff or alert. A standing health check monitors DB, third-party API, and auth connectivity so the app flags or resets itself before anyone notices it's degraded.

def handle_activity_error(exc: Exception, item_id: str) -> None:
    if isinstance(exc, BusinessException):
        # automation can't make the call, needs a human
        review_queue.enqueue(item_id, reason=str(exc))
        return

    if isinstance(exc, SystemException):
        # transient failure, safe to retry
        if attempt_count(item_id) < MAX_RETRIES:
            retry_queue.enqueue(item_id, delay=backoff(item_id))
        else:
            alerting.notify(f"system exception exhausted retries: {item_id}")
        return

    raise exc  # anything unclassified fails loud, on purpose

RPA-to-AI Migration

Modernizing legacy RPA into LLM-assisted automation

Problem

A long-running RPA program had dozens of bots that were great at structured, rules-based work and fell apart on anything needing judgment: unstructured documents, ambiguous free text, exceptions the decision tree wasn't built for.

Approach

Instead of ripping it all out, I scoped a migration that layered LLM reasoning on top of the existing automation — only for the tasks where rules-based logic was actually breaking — and left the reliable bots doing what they were already good at. The real work was deciding exactly where deterministic automation should end and AI judgment should start, then building clean interfaces between the two so the whole pipeline stayed auditable.

ApproachHybrid RPA + LLM architecture, scoped modernization plan
StackBlue Prism, Python, LLM APIs, document/text processing
OutcomeExtended automation coverage into previously manual exception handling
Technical Integration Proposal

Keeping Jira the single source of truth across three systems

Context

A proposal I built for an interview exercise — not delivered client work — but close to how I'd actually approach it, so it's worth showing in full.

Problem

A customer wanted to hand global hardware procurement and lifecycle management to a new IT platform, but their Jira Asset Management instance had to stay the authoritative record auditors query — and it couldn't step on the Mac and Windows device-management tools already live.

Approach

I proposed an architecture where the new platform drives procurement and lifecycle events, the device tools report real-time health, and Jira reconciles all three into one conflict-resolved record. It included a full field-mapping spec, explicit conflict-resolution rules (a stale device check-in can't overrule an active repair status), a PII-stripping layer before anything writes to Jira, and a phased rollout that starts read-only and upgrades to real-time sync without re-architecting later. I also built a small local mock server to stress-test my mapping and conflict assumptions against a simulated API before presenting — rather than showing a diagram I hadn't tested.

ApproachMulti-system reconciliation architecture, field mapping, conflict-resolution rules, phased rollout plan
StackREST APIs, delta polling, webhook-based event sync, PII-filtering middleware
OutcomeA complete, defensible technical proposal, self-validated against a mocked API, not a shipped integration
